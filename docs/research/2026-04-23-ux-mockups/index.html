When verified arrives,
what does the user feel?

Fast lane renders your graph at t≈1s. The verified lane arrives at t≈45s and may add edges, remove wrong ones, or upgrade inferred edges to verified. How should that moment land? Four UX patterns below — each is a working interactive mockup. Click through to feel the difference.

Simulate

Strengths

  • Minimal screen real estate — doesn't interrupt
  • Familiar: GitHub PR-badge mental model
  • Fastest to implement in v1

Risks

  • Most users won't click the expander
  • They'll see graph change — won't know which edges
  • "Silent truth swap" concern only partially mitigated

Implementation

  • Badge component + diff-summary chip
  • Collapsible panel with grouped list
  • No animation logic required
B

Animated diff-first render

Force 2 seconds of attention. Make the change impossible to miss.

Simulate

Strengths

  • Impossible to miss — user sees exactly what changed
  • Restores user's mental model forcibly
  • Visually satisfying at small graph sizes

Risks

  • Annoying on every save-triggered update
  • Noise on 200+ node graphs
  • Animation + SVG-diff code to build & test

Implementation

  • SVG edge-diff tween engine
  • 2s animation budget per transition
  • Replay button that re-runs sequence
C

Pending apply

The user is the gate. Truth changes only when they click.

verified pass ready

Simulate

Strengths

  • Strictly aligned with "must be verifiable" philosophy
  • No silent truth mutation — ever
  • User knows exactly when the model changed

Risks

  • Friction on every verified run
  • Users will develop "click-through" fatigue
  • If always clicked, reduces to Option A with extra step

Implementation

  • Pending state in the state machine
  • Non-blocking notification card
  • Dismissal → badge remains clickable
